● Zero errors. Here's what changed and why:

  Supabase fallback was always returning null because the supabase client uses
  only the anon key, which RLS blocks for shop-specific rows. The fix uses the
  stored backend JWT as the Authorization: Bearer header in a direct PostgREST
  REST call. Since the backend JWT is typically signed with the same Supabase
  JWT secret, RLS will evaluate auth.jwt() claims and grant access to the shop's
   rows.

  Product list was capped at 200 — if a shop has more than 200 products, scanned
   barcodes for products past position 200 would never be found in the local
  cache. Raised the limit to 1000.

  The log will now show:
  - [Scan] Supabase REST result: <uuid> | status: 200 — product found via
  fallback
  - or [Scan] Supabase REST result: null | status: 200 — truly not in DB → add
  new product
